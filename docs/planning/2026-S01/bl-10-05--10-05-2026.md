# Sprint 1 — 10 May 2026

Senior AI/ML Engineer take-home build. 2-hour budget. Brief grades a *small thing done well*, trade-offs named in the writeup.

> Source materials: [docs/plan.md](../../plan.md), [docs/style-guide.md](../../style-guide.md), [CLAUDE.md](../../../CLAUDE.md).

## What the tool does

A CLI that points at a public GitHub repo URL and produces a Monday-morning brief covering two sections:

- **§1 — This week's new issues**: opened in the past 7 days, still open. LLM categorises → prioritises → summarises.
- **§2 — FYI: ongoing activity**: open issues older than 7 days that gained new comments in the past 7 days. Same pipeline plus a "what's new this week" interpretation that reads the new comments.

Outputs land in `reports/<date>/<owner>__<repo>/`:

- `brief.md` — Markdown for reading
- `brief.json` — canonical structured payload (downstream-friendly)
- `brief.html` — single-file styled report
- `run.json` — reproducibility snapshot (provider, model, prompt versions, tokens, duration, any warnings)

## Key design decisions (defended in the writeup)

- **Workflow, not agent.** Fixed pipeline, structured output at every stage, auditable end-to-end.
- **Provider abstraction with an ABC.** Ships with Gemini + Ollama. Adding a third is one new file under `src/issue_triage/providers/` plus an entry in the factory — no surgery.
- **Prompts as markdown files.** A non-engineer can iterate on LLM behaviour without touching Python.
- **Nothing hardcoded that should be the user's call.** Provider name, model name, and category taxonomy all live in `config.yaml`. `maintainer_context.md` (editable, ships with sensible defaults) is injected into the system prompt.
- **Bounded by transparency, not truncation.** Tool processes every issue meeting the time-window filter; cost is bounded by a pre-flight estimate that prints expected calls / tokens / cost / duration and waits for `[y/N]`. `--yes` skips for CI; `--max-cost <$>` adds a hard ceiling. A safety-net cap (`safety_max_issues`, default 1000) only fires for pathological inputs.
- **External content treated as untrusted.** Issue text wrapped in `<issue>...</issue>` delimiters before insertion into prompts. HTML render escapes every dynamic value. URL `href`s scheme-allowlisted. CSP meta tag for defence in depth.
- **No silent skips.** When something fails (parse error, malformed LLM output, §2 follow-up call failure), the issue stays in the brief with a clear marker, never disappears. Counts of failures land in `run.json`.
- **Credentials never leak.** API keys are in private attributes with `repr=False`; a logging filter scrubs known env-var values from log records; retry logs contain only URL + status.

## Tests

Tests are written to cover risky pathes:

- URL parsing rejects malicious inputs (already done in T1.1)
- Pydantic config loads + validates (already done in T1.1)
- Pipeline produces a valid `Brief` end-to-end with a `FakeProvider`
- HTML render escapes XSS attempts
- Provider factory dispatches the right class

Manual smoke testing with **both providers across 2–3 real public repos** is the primary validation. The brief grades "can a reviewer clone this and run it" — a working tool against real data is worth more than extensive unit tests on hypothetical inputs.

## Non-goals (named in the writeup)

- Reading repository source code or suggesting code changes
- Scheduled automation (no GitHub Action, no cron)
- Multi-repo support, trend tracking
- Posting back to GitHub (auto-labelling, commenting)
- Reactions as a §2 trigger — GitHub's API doesn't filter reactions by `updated_at`. Comments are the §2 signal in v1; reactions still inform priority weight.

## Cut order if running tight

A decision tool, not a stopwatch. If something has to give:

1. **Cut T1.7 (HTML render)** — Markdown + JSON is enough to demonstrate the design. Note in writeup: "the renderer interface is in place; HTML is one new function."
2. **Cut T1.5 themes aggregation** — keep per-issue priority + summary; drop the cross-issue list.
3. **Cut T1.8 mid-run guard** — keep the pre-flight estimate (the visible cost story); drop the runtime overrun guard.

**Never cut T1.9.** The brief grades the writeup heavily.

## Pre-submission checklist

- No secrets in `git log -p | Select-String -Pattern "key|token|secret"`
- README runs cleanly on a fresh clone
- Tool produces brief.md / brief.json / brief.html / run.json against a small public repo
- Both providers tested end-to-end (Gemini + Ollama)
- `pytest` passes
- No bare `except` anywhere
- Helpful single-line errors on common failures (bad URL, missing key, no Ollama, rate-limited)
